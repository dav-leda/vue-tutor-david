El problema es que __el plan gratuito de MockAPI s√≥lo permite crear 2 recursos, y ya usamos uno para usuarios y otro para productos__ üò¨Ô∏è

La soluci√≥n es que dentro del recurso de usuarios, en el objeto de cada usuario haya __un array de pedidos (orders)__ (al final de esta p√°gina pueden ver un ejemplo de c√≥mo les quedar√≠a el JSON de usuarios en MockAPI con este array de pedidos).

Para formatear la fecha (timestamp) pueden usar esta _helper function_ y ponerla en un archivo `.js` aparte (la convenci√≥n es poner este tipo de funciones en un directorio llamado `/utils` dentro de `/src`):

  minute: 'numeric'

Otra opci√≥n es guardar en el backend la fecha en bruto (`new Date()`) y luego formatearla en el momento de mostrarla.

Pero recuerden que para agregar un nuevo pedido dentro de este array __no pueden usar POST, deben usar PUT__. Si usan POST estar√≠an generando un nuevo usuario, pero lo que deben hacer es modificar una propiedad dentro del objeto del usuario (o sea, la propiedad __orders__).

Ser√≠a bueno que para chequear si existe un usuario con el nombre (o email) y passwords ingresados por el usuario __no hagan este chequeo en el frontend con *find()*__ ‚ö†Ô∏è

Esto es porque traer al frontend __todos los datos de todos los usuarios incluyendo sus passwords__ es considerado una mala pr√°ctica, porque de esta forma cualquier usuario que sepa usar las Dev Tools del browser __podr√≠a tener acceso a todos los passwords de todos los usuarios__ ü§¶

La √∫nica informaci√≥n que deber√≠a llegarle al frontend son los datos del usuario que est√° intentando hacer Login, __no de todos los usuarios__. Para poder pedirle a MockAPI √∫nicamente este dato deben usar [search params](https://github.com/mockapi-io/docs/wiki/Code-examples#filtering). 

Tambi√©n se puede hacer esto mismo usando la sintaxis para [query strings](https://www.semrush.com/blog/url-parameters/), con un `?` luego del enpoint, luego el nombre de la propiedad a buscar (en este caso es `username`), luego un `=` y luego el dato que se quiere buscar (en este caso, lo que ingres√≥ el usuario en el formulario de Login, o sea, `this.form.username`):

```js

```

MockAPI siempre retorna un array, y si no encuentra nada, retorna un array vac√≠o. Por eso la respuesta al _query_ es el pimer elemento del array (`res[0]`). 

Entonces, el _method_ para el Login podr√≠a quedar as√≠:

  } else if (user.password !== this.form.password) {

  } else {
    // Por ahora guardar el usuario en data
    // luego debe ser guardado en Vuex o Pinia
    this.user = user
    
    this.form.resetForm()
    
    // Si van a guardar el usuario en localStorage
    // para persistir la sesi√≥n ser√≠a bueno
